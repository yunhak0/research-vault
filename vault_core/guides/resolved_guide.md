# Resolved 문서 작성 가이드

## 1. 목적

문제(PRB)를 **해결한 접근**을 상세히 기록하는 문서.
"어떻게 해결했고, 왜 효과적인가?"에 답한다.
논문의 Methods/Results 섹션 작성 시 직접 참조하는 핵심 문서.

## 2. ID 규칙

```
RES-[NNN]
```
RES 번호는 PRB 내에서만 유일하면 된다.

## 3. 파일 위치

```
[PROJECT]/problems/PRB-[NNN]_xxx/resolved/RES-[NNN]_[name].md
```

## 4. RES 문서 생성 조건

- INV에서 시도한 접근이 **문제를 해결**했을 때
- 문제의 근본 원인이 파악되고 **검증된 해결책**이 있을 때

## 5. 섹션별 작성 규칙

### 5.1 해결책 — `> **해결책**: ...`로 시작하는 한 문장.
### 5.2 해결책 요약 — 무엇을 했는가 + Before/After 비교 테이블
### 5.3 근거 — 문제 진단 + 해결 가설 + 왜 효과적인가
### 5.4 실험 결과 — 정량적 결과 테이블 필수. W&B Run ID 참조.
### 5.5 검증 — 재현성 + Overfitting 확인 + 한계 및 조건
### 5.6 후속 영향 — 프로젝트 다른 부분에 미치는 영향
### 5.7 논문 반영 사항 — Figure 후보 + 핵심 수치 + Methods/Results 기술

## 6. Status 정의

| 상태 | 정의 |
|------|------|
| Completed | 해결책 기록 완료 |
| Validated | 추가 검증 완료, Foundation 승격 후보 |

## 7. RES → FND 승격 기준

RES가 아래 조건을 모두 만족하면 Foundation으로 승격 가능:

1. **재현성 확인**: 다양한 조건에서 일관된 결과
2. **프로젝트 핵심 기반**: 이후 작업의 전제가 됨
3. **변경 가능성 낮음**: 향후 변경 시 대규모 영향

승격 시 `foundations/`에 FND 문서를 생성하고, 원본 RES의 `legacy_ids`에 기록한다.

## 8. analyze/ 연동

RES 문서는 핵심 수치를 생산한 스크립트를 명시적으로 연결하여 **재현성**을 보장한다.

> **스크립트 명명·docstring·재현성 규칙**은 프로젝트 `.CLAUDE.md`의 "analyze/ Convention" 섹션을 참조.
> 여기서는 **문서 작성 시** analyze/를 어떻게 참조하는지만 정의한다.

### RES 문서에서의 참조

- frontmatter `analyze_scripts`에 해당 해결책의 결과를 재현할 수 있는 스크립트 경로 기록
- "재현 스크립트" 섹션에 각 결과와 스크립트의 대응 관계 기록
- **목표**: "이 스크립트를 실행하면 이 결과를 재현할 수 있다"가 명확해야 함
- INV에서 만든 스크립트가 RES에서도 유효하면 `analyze_scripts`에 그대로 기록

## 9. Notion 아카이빙

RES는 개별 아카이빙하지 않는다. 핵심 내용은 _overview.md를 통해 아카이빙됨.
단, FND로 승격된 경우 FND 문서가 아카이빙된다.
