# Investigating 문서 작성 가이드

## 1. 목적

문제(PRB)에 대해 **현재 진행 중인 조사, 진단, 실험**을 기록한다.
기존의 실험 설계서(EXP)와 실험 로그(LOG)의 역할을 통합한 문서.
"무엇을 시도하고 있고, 어디까지 왔는가?"에 답한다.

## 2. ID 규칙

```
INV-[NNN]
```
INV 번호는 PRB 내에서만 유일하면 된다.

## 3. 파일 위치

```
[PROJECT]/problems/PRB-[NNN]_xxx/investigating/INV-[NNN]_[name].md
```

## 4. 기존 EXP+LOG와의 차이

| 항목 | 기존 (EXP+LOG) | 현재 (INV) |
|------|----------------|------------|
| 문서 수 | 설계 1개 + 로그 1개 = 2개 | 1개로 통합 |
| 범위 | 단일 실험 | 진단 과정 전체 (여러 실험 포함 가능) |
| 맥락 | 독립적 | 항상 PRB에 소속 |
| 업데이트 | LOG는 실시간 기록 | 핵심 발견 중심으로 업데이트 |

## 5. 섹션별 작성 규칙

### 5.1 현황 요약 — 1-2문장. 목표와 현재 진행 상태.
### 5.2 문제 정의 — 관찰된 현상 + 정량적 증거 (메트릭 테이블)
### 5.3 가설 — 테이블 (# | 가설 | 검증 방법 | 상태: 미검증/지지/기각)
### 5.4 진단 계획 — 단계별(Phase) 계획
### 5.5 검증 계획 — 실험 전에 반드시 작성

**핵심 원칙**: "실험을 시작하기 전에, 어떤 결과가 나오면 성공/실패인지 미리 정의한다."

**성공/실패 판단 기준**:
- 정량적으로 측정 가능해야 함 ("좋아졌다" ✖ → "Pearson > 0.85" ✔)
- 성공 조건과 실패 조건을 둘 다 명시

**기대 수치**:
- Baseline(현재값)은 실측치, Target(기대값)은 근거 기반 예측
- 근거: 선행 연구, 이론적 분석, 또는 preliminary 실험 결과

**검증 단계 (Pre/Post)**:
- 빠른 검증(Pre): 본 실험 전 1일 이내에 확인할 수 있는 사전 테스트 (예: 소규모 데이터셋, sanity check)
- 본 실험: 핵심 실험 설계 요약
- 교차 검증(Post): 결과의 신뢰성 확인 (예: 다른 데이터셋, ablation, 통계 검정)

### 5.6 방법론 — 데이터, 모델, 파라미터, 평가 지표
### 5.7 중간 결과 — 발견 중심으로 구성 (날짜별이 아님)
### 5.8 해석 및 인사이트
### 5.9 다음 단계 — 체크리스트 형태

## 6. Status 정의

| 상태 | 정의 | 다음 단계 |
|------|------|----------|
| Draft | 계획 수립 중 | 실험 시작 |
| In Progress | 조사/실험 진행 중 | 결과 기록 |
| Completed | 조사 완료, 결론 도출 | LRN 또는 RES 생성 |

## 7. INV 완료 후 전환

- **가설 기각 / 접근 실패** → `learned/`에 LRN 문서 생성
- **문제 해결** → `resolved/`에 RES 문서 생성
- **추가 조사 필요** → 새 INV 문서 생성

전환 시 _overview.md의 "핵심 발견"과 "시도한 접근"을 업데이트한다.

## 8. analyze/ 연동

INV 문서에서 분석 스크립트를 사용할 때는 docs↔analyze 간 양방향 참조를 유지한다.

> **스크립트 명명·docstring·재현성 규칙**은 프로젝트 `.CLAUDE.md`의 "analyze/ Convention" 섹션을 참조.
> 여기서는 **문서 작성 시** analyze/를 어떻게 참조하는지만 정의한다.

### INV 문서에서의 참조

1. frontmatter `analyze_scripts`에 관련 스크립트 경로 기록
2. "분석 스크립트" 섹션에 각 스크립트의 용도와 실행 환경 기록

## 9. Notion 아카이빙

INV는 개별 아카이빙하지 않는다. 핵심 내용은 _overview.md를 통해 아카이빙됨.
