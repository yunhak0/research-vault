# Foundation 문서 작성 가이드

## 1. 목적

프로젝트의 **확립된 기술적 토대**를 기록하는 문서.
실험으로 검증되었고, 앞으로 변경 가능성이 낮은 설계 결정을 다룬다.
"우리는 왜 이렇게 하는가?"에 대한 답을 제공한다.

## 2. ID 규칙

```
FND-[NNN]

예시: FND-001
```

## 3. 파일 위치

```
[PROJECT]/foundations/FND-[NNN]_[name].md
```

## 4. Foundation이 되는 조건

아래 조건을 모두 만족해야 한다:

1. **실험적 검증 완료**: 가설 → 실험 → 결과로 확인됨
2. **안정성 확인**: 현재 성능의 핵심 기반으로 작동 중
3. **변경 가능성 낮음**: 향후 변경 시 대규모 리팩토링 필요

**Foundation이 아닌 것:**
- 아직 실험으로 검증되지 않은 아이디어 → INV
- 실패한 시도 → LRN
- 하이퍼파라미터 튜닝 결과 → RES 또는 INV

## 5. 섹션별 작성 규칙

### 5.1 제목 — 기술 결정의 핵심을 드러내는 명사구. 20자 이내.
### 5.2 핵심 결정 — `> **핵심 결정**: ...`로 시작하는 한 문장.
### 5.3 배경 — 문제 인식 + 기존 접근의 한계
### 5.4 핵심 아이디어 — What / How / Why
### 5.5 실험적 검증 — Before/After 비교 테이블 필수. W&B Run ID 참조 권장.
### 5.6 설계 결정 사항 — 테이블 (결정 | 선택 | 대안 | 선택 이유)
### 5.7 한계 및 열린 질문 — 재검토 조건 명시

## 6. Status 정의

| 상태 | 정의 |
|------|------|
| Draft | 초안 작성 중, 근거 정리 중 |
| Validated | 실험적 검증 완료, 프로젝트 토대로 확립 |

## 7. Foundation 생성 경로

1. **PRB → RES → FND**: 문제 해결 과정에서 확립된 설계가 Foundation으로 승격
2. **직접 생성**: 프로젝트 초기 설계 결정을 기록

승격 시 `legacy_ids`에 원본 문서 ID를 기록한다.

## 8. Notion 아카이빙

- **아카이빙 대상**: `status: Validated`
- **Type**: FND
- Foundation은 프로젝트의 핵심 레퍼런스이므로 아카이빙 우선순위 높음
